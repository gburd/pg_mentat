# Mentat to PostgreSQL Extension Migration Plan

## Context

Mentat is a Datomic-inspired embedded datalog database written in Rust, currently built on SQLite. The goal is to transform it into a production-ready PostgreSQL extension (pg_mentat) with a standalone network frontend (mentatd) that speaks the Datomic wire protocol.

**Why this change matters:**
- **Scalability**: PostgreSQL's robust transaction model and concurrency controls will enable multi-user, production-scale deployments
- **Ecosystem integration**: pg_mentat will make datalog queries available to any PostgreSQL-connected application
- **Datomic compatibility**: mentatd will allow existing Datomic client libraries to work with PostgreSQL
- **WASM extensibility**: Dynamic function loading will enable JVM-free extension of database capabilities

**Current State:**
- 21-crate Rust workspace, ~6,097 lines of tests, comprehensive CI/CD
- SQLite-based storage with WAL mode, FTS4 fulltext, custom indexes
- Implements core Datomic features: Datalog queries, pull API, transactions, schema management
- Missing: Aggregate functions, rules, time-travel queries, transaction functions, remote API

**Key Technical Challenges:**
1. Tight coupling to rusqlite and SQLite-specific features (WAL, FTS4, pragmas)
2. Large monolithic modules (db.rs is 143KB)
3. No existing storage abstraction layer
4. No network protocol or server interface
5. Dependencies on deprecated tooling (actions-rs in CI)

**Custom EDN Type Implementation:**
```rust
#[derive(PostgresType, Debug, Clone)]
#[pgvarlena_inoutfuncs]
pub struct EdnValue {
    inner: Vec<u8>,  // Binary encoded EDN
}

impl PgVarlenaInOutFuncs for EdnValue {
    fn input(input: &CStr) -> Self {
        // Parse EDN text → binary
        let edn = edn::parse::value(input.to_str().unwrap()).unwrap();
        let binary = Self::encode(&edn);
        EdnValue { inner: binary }
    }

    fn output(&self, buffer: &mut StringInfo) {
        // Binary → EDN text
        let edn = Self::decode(&self.inner);
        buffer.push_str(&format!("{}", edn));
    }
}

// Binary encoding using rmp-serde (MessagePack)
impl EdnValue {
    fn encode(value: &edn::Value) -> Vec<u8> {
        rmp_serde::to_vec(value).unwrap()
    }

    fn decode(bytes: &[u8]) -> edn::Value {
        rmp_serde::from_slice(bytes).unwrap()
    }
}
```

## Design Decisions (Confirmed)

1. **Backend Strategy**: PostgreSQL only - no SQLite compatibility layer. Cleaner implementation, focused effort. Users migrate via export/import tool.
2. **WASM Runtime**: wasmer - production-ready, excellent security sandbox, good PostgreSQL integration examples
3. **EDN Storage**: Custom PostgreSQL type via pgrx - first-class type integration for optimal performance and PostgreSQL ecosystem compatibility
4. **Datomic Protocol**: Reverse-engineer from Datomic client JAR. Goal: existing Datomic applications work unchanged except for configuration pointing to mentatd instead of Datomic servers.
5. **Transaction Metadata**: Use hybrid approach - PostgreSQL MVCC for visibility, mentat tables for datalog semantics (transaction-as-entity)
6. **Architecture**: mentatd (frontend) speaks Datomic protocol → translates to SQL → calls pg_mentat functions → PostgreSQL storage

## Datomic Protocol Reverse Engineering Tools & Process

**Tools Required:**
- **cfr** or **procyon** - Java bytecode decompiler
- **Wireshark** with Transit/EDN dissectors (custom if needed)
- **mitmproxy** - HTTP/HTTPS protocol inspection
- **datomic-client** JAR - Official client library to analyze
- **Datomic free/dev** - Reference implementation for testing

**Process:**
1. **Extract and decompile Datomic client JAR:**
   ```bash
   unzip datomic-client.jar -d datomic-src/
   cfr datomic-src/**/*.class > decompiled.java
   ```

2. **Identify key classes:**
   - Connection/client initialization
   - Query submission (datalog, pull, entity)
   - Transaction submission
   - Result deserialization
   - Network transport layer

3. **Network protocol analysis:**
   - Run Datomic free locally
   - Write simple test application using Datomic client
   - Capture traffic with Wireshark/tcpdump
   - Analyze request/response format
   - Document serialization format (Transit, EDN, Fressian?)

4. **Protocol specification:**
   - Document message formats
   - Create protocol buffer/schema definitions
   - Build test corpus of known queries/transactions
   - Implement protocol parser in Rust

5. **Incremental implementation:**
   - Start with simplest operations (connect, simple query)
   - Add transaction support
   - Add pull API
   - Add advanced query features
   - Test with real Datomic client

**Expected Protocol Characteristics:**
- **Transport**: Likely HTTP/HTTPS (REST-like) or custom TCP
- **Serialization**: Transit (JSON/MessagePack variant) or Fressian (Datomic's binary format)
- **Operations**: connect, query, transact, pull, entity, db-as-of, db-since, sync-schema
- **Authentication**: API keys or tokens
- **Streaming**: Possible chunked responses for large result sets

**Fallback Strategy:**
- If protocol is too complex or undocumented: implement subset supporting most common operations
- Document limitations clearly
- Provide direct pg_mentat SQL API as alternative
- Consider contributing to open-source Datomic client with mentat support
